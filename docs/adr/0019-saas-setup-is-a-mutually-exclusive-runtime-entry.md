# SaaS setup is a mutually exclusive tenant runtime entry

For hosted SaaS, the fenced provisioner owns tenant application lifecycle and selects one of two entry points from the same published runtime artifact: `node dist/cli/setup.js` for setup, or the existing CLI-installed Flue server for operate. Setup resolves only its provisioner-owned boot environment at `apps/runtime/src/setup-server.ts`, then passes a typed setup boot into the narrow Hono application before any Flue operate module is imported. This supersedes ADR 0010's CLI-only composition-root rule only for the hosted setup profile; self-hosted lifecycle and hosted operate keep the existing CLI-owned typed dependency handoff.

The entries are mutually exclusive restarts of one leased tenant application, never concurrent supervisors or replacement applications. Setup owns one WhatsApp account and the same tenant credential store and local Conversation Archive as operate, while omitting the Managed Chat Inbox, Coalescer, Speaker, GitHub, model, Flue agent, workflow, and channel composition. The unauthenticated health route never contains pairing material; pairing and chat enumeration remain HMAC-authenticated bridge capabilities.
