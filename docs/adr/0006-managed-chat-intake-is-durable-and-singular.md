# Managed Chat intake is durable and singular

> Superseded by [ADR 0014](./0014-window-delivery-is-an-at-least-once-wake.md): the admission-receipt state machine and manual Uncertain resolution described below are replaced by the at-least-once pending→done/failed Window delivery.

One WhatsApp intake path replaces the separate persistence and coalescer listeners. It first appends every message observed by or sent through the configured account to the Conversation Archive, regardless of chat, and only then applies the Managed Chat decision; accepted archive entries enter the Managed Chat Inbox as pending and feed the lossless Coalescer, which gives every Window a stable application identity. The relay records that it is dispatching before calling Flue, records the returned `dispatchId` and `acceptedAt` as the durable Admission Receipt, and only then settles the Window as admitted. Pending work that has never crossed the seam is replayed at startup, while a crash or unresolved error after dispatch begins leaves the Window Uncertain and never triggers an automatic duplicate; `doctor` attempts read-only reconciliation and requires an explicit resolution when absence cannot prove non-delivery. A file-backed Flue persistence adapter owns the accepted submission and all processing recovery after the receipt, while the application database owns the archive, inbox, relay state, and pre-admission recovery.

Verified GitHub webhook ingress applies the same seam semantics with the provider delivery ID as its application identity. The application database records `received` before interpretation, `dispatching` with the explicit repository-to-Managed-Chat route before calling Flue, and the complete `dispatchId` plus `acceptedAt` receipt afterward. A replacement process may resume only a delivery still proven `received`; persisted `dispatching` becomes Uncertain at startup, while active-process and terminal redeliveries are deduplicated. This is a local process-replacement boundary, not a PID, stale-lock, or cross-host recovery protocol.
