# One shared GitHub App set, routed by installation id

The product operates exactly three GitHub App identities — `coder`, `reviewer`, and `planner` (`packages/installation/src/schema.ts:14`) — created once by the operator and installed by every tenant organization. Tenancy is expressed by GitHub's own installation mechanism: each org's installation carries an installation id, every webhook names the installation that sent it, and delivery routing is a lookup from installation id to tenant row. Installation-scoped tokens mean one tenant's credential can never reach another tenant's repositories. The provisioner-era design of minting a fresh App set per tenant is retired; no code path may create GitHub Apps at runtime.

Three identities rather than one is load-bearing, not ceremony: GitHub refuses to count a pull-request approval authored by the PR's own author, so the reviewing identity must differ from the authoring identity for automated review to produce real approvals. The `planner` App continues to carry the webhook secret as the single delivery entry point. The multi-app ingress ledger keyed by `github_app_id` remains correct under this topology with a fixed key domain of three.
