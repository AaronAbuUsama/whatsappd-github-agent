# The operator admin web app is the management surface

Tenants are provisioned and managed through `apps/web`, signed into by the operator alone. Creating a coworker, pairing its WhatsApp, selecting its managed chats, applying its GitHub configuration, watching its health and restarting it are web operations, not CLI operations. This supersedes ADR 0023 in one respect only: `apps/web`, `packages/auth`'s sign-in, and the oRPC contract at `packages/api/src/routers/index.ts` are kept and rewired rather than demolished. ADR 0023's central decision ŌĆö one Node process holding every tenant, tenants as rows rather than deployments ŌĆö stands unchanged, as do ADRs 0020, 0021 and 0022's per-tenant directory layout.

ADR 0023 fused two separable decisions and only one of them was intended. Retiring the Dokploy container-per-tenant provisioner is correct and is where the fenced leases, health probes, Swarm service names and the unrunnable hosted Reviewer all came from; `apps/api/src/provisioner.ts`, `provisioner-providers.ts`, `provisioner-hosted.ts` and `tenant-bridge.ts` are still demolished. Deleting the management UI was not intended and was never authorized: the oRPC verbs the wizard already calls ŌĆö `coworker.create`, `ensureSetup`, `whatsapp.pairing`, `chats.list`, `activate`, `github.apply` ŌĆö contain no provider concept and name the right operations. What changes beneath them is the implementation: `packages/api/src/coworker.ts` creates a tenant row and its directory inside the running service instead of calling out to Dokploy and Turso and waiting on a container.

Those verbs are nonetheless the wrong *shape*, and this decision requires changing them. Every one is `x(userId)` and tenant resolution is `WHERE tenant.user_id = ?` (`packages/api/src/coworker.ts:158-192, 300-306`): the contract encodes user-equals-tenant, which is correct for a SaaS customer with one coworker and incompatible with an operator holding several. The schema says the same thing four times over ŌĆö `tenant_user_unique`, `tenant_subscription_unique`, `subscription_entitlement_user_unique`, and a `NOT NULL` entitlement foreign key. An operator admin surface therefore requires a `tenantId` parameter on all fourteen verbs, a `coworker.list`, tenant-scoped routes in the web app, and those four constraints dropped. A tenant list and switcher are new interface, not a rewiring: the onboarding page redirects to the dashboard as soon as the first tenant is active, so today there is no route to a second one.

Public self-serve signup and Polar billing stay switched off while the only tenants are the operator's own organizations, and return when someone who is not the operator needs an account ŌĆö the seam being preserved is that every tenant-specific thing hangs off the tenant row and its directory, so the sign-up surface is additive when it arrives. This qualifies ADR 0022's "no control-plane database": an authenticated web app must persist its operator identity and its tenant rows, so one small store for users and tenants survives alongside the per-tenant SQLite directories it points at.
