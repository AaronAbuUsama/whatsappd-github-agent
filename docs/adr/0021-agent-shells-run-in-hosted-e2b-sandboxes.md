# Agent shells run in hosted E2B sandboxes

Model-directed shell work — the Coder's implement/verify loop and the Reviewer's exercise runs — executes in per-job E2B Firecracker sandboxes behind Flue's `SandboxFactory` seam, created at job start and destroyed at job end. This replaces both `local()` for the Coder (`apps/runtime/src/app.ts:60`), which Flue's sandbox guide explicitly forbids as an isolation boundary for untrusted or multi-tenant work, and the host-socket Docker runner (`packages/installation/src/reviewer-docker-sandbox.ts`) as the Reviewer's isolation mechanism. Nothing in the runtime container mounts the Docker socket; a sandbox escape therefore lands inside a disposable provider micro-VM, not on the host that runs tenant sessions and secrets.

The adapter follows Flue's documented E2B blueprint (`docs/reference/flue/docs-ecosystem-sandboxes-e2b.md`): the application creates and configures the E2B sandbox — template, timeout, network policy — and hands it to the agent; E2B's `rm` lacks recursive/force semantics, which is irrelevant because per-job sandboxes are discarded rather than cleaned. The E2B API key is tenant-independent operator configuration and lives in the secret store, never in tenant data. Job budget remains bounded by the existing shell timeout; provider cost at dogfood volume is negligible and is accepted.
